How This System Works

This system does not track individual animals. Instead, it models environmental conditions — vegetation health, water availability, terrain accessibility, and community-reported conditions — that strongly predict where pastoral herds will move. When multiple environmental signals agree, the system generates a movement corridor forecast classified as High, Moderate, or Low likelihood. The logic mirrors how pastoralists actually make decisions: animals move toward resources, away from scarcity.

How Classifications Are Determined

Instead of arbitrary percentages, this system uses a 3-tier classification based on signal agreement. Here is the decision logic:

OUTPUT

Mixed signals — some indicators strong, others unavailable or moderate

Weak environmental stress, resources adequate, no community confirmation

Why not percentages?

Using a percentage like "89% likelihood" implies a precision that doesn't exist in this type of environmental modeling. A classification like High is more honest — it communicates that multiple independent signals agree this corridor is active, without falsely implying exact quantitative certainty. Precision should match the actual quality of the data.

About the Model

A complete explanation of how this system works, why it is effective, and what it does and does not claim to do.

The Core Insight

Pastoral herd movements are not random. They are environmentally determined decisions. Pastoralists have millennia of knowledge about which environmental signals — water, vegetation, terrain, risk — should prompt movement and in which direction. This system works because it operationalizes that same decision logic using satellite remote sensing. It models the environment that drives the decision, not the decision itself.

How the System Works — Step by Step
1

Satellite Data Detects Environmental Conditions

Satellites observe vegetation health (NDVI), water body extent, soil moisture, terrain slope, and flood extent across the target region. These signals change over days and weeks and directly reflect whether an area can support a herd.

VIIRS · Sentinel-1 · SMAP · GPM IMERG
2

Accessibility Modeling Evaluates Viable Routes

Not all terrain is traversable. A GIS processing layer evaluates which areas are physically accessible — accounting for slope, flooding, road infrastructure, and known barriers. This produces a movement feasibility surface across the region.

GIS · DEM Analysis · Flood Masking
3

Community Reporting Adds Real-World Confirmation

Pastoralist communities, field monitors, and local NGO workers submit structured field reports via mobile or radio. These reports confirm or contradict what satellite data suggests — providing ground truth that satellites cannot capture (e.g., a local conflict, a broken water pump, a newly opened gate).

Community Network · Field Reports
4

Rule-Based Environmental Scoring Generates Likelihood

A scoring engine evaluates each potential corridor based on: how strong each environmental signal is, whether signals agree or conflict, and whether community reports confirm the pattern. The result is a High / Moderate / Low classification — not a fake percentage. Each classification is fully explainable from the signal inputs.

Rule Engine · Signal Weighting · Decision Logic
5

System Outputs Probabilistic Movement Corridors

The final output is a map of probable movement corridors — spatial paths where herd movement is environmentally motivated. These corridors help humanitarian organizations pre-position resources, prevent conflicts, and plan interventions. Individual animals are never tracked.

Future: ML model trained on historical movement patterns
Why This System Works Well
  • Environmental signals are strong predictors. Decades of pastoral research confirm that herds move toward water and vegetation. These signals are reliable and measurable from space.
  • Multi-source agreement reduces error. When NDVI, water availability, soil moisture, and community reports all point the same direction, confidence is high. Disagreement between sources is itself informative — it flags uncertainty honestly.
  • Community reporting fills satellite blind spots. Satellites can't see a broken borehole or a tribe conflict blocking a route. Community reports catch what remote sensing misses.
  • Tracking decisions, not animals, is more scalable. Environmental data covers entire regions simultaneously. You don't need a sensor on every animal — you need to understand what drives the decision.
  • Rule-based logic is transparent and auditable. Unlike a black-box ML model, every classification can be traced back to specific signal inputs. This matters for humanitarian accountability.
  • The system improves with ground truth data. Every confirmed movement adds to the historical record. Future ML integration will use this data to improve predictions over time.
What This System Claims
  • That environmental conditions predict movement direction
  • That multi-signal agreement increases forecast reliability
  • That corridor classifications are explainable from inputs
  • That community reports improve forecast accuracy
  • That this can be operationalized with backend data integration
What This System Does NOT Claim
  • It does not track individual animals
  • It does not generate precise timing predictions
  • It does not replace local pastoral knowledge
  • It does not work without quality satellite data inputs
  • It does not provide exact percentages — classifications are qualitative

Explainability

Understanding every classification decision — the signals, the logic, and the limitations.

Why Explainability Matters

In humanitarian applications, a forecast that cannot be explained is a forecast that cannot be trusted. Every output from this system must be traceable back to specific observable inputs. If a signal is unavailable, that uncertainty must be visible — not hidden behind a confident-looking number. This section documents the full decision logic of the classification engine.

Signal Weighting Logic

Each signal contributes differently to the corridor classification. The relative importance of signals depends on which are available and their agreement.

👥Community Reports
Highest weight. Directly observational. Overrides satellite-only signals if confirmed. A confirmed community report of movement initiation is the strongest possible signal.
🌿Vegetation (NDVI)
Very high weight. The most reliable satellite predictor. NDVI decline below a threshold consistently precedes movement in documented pastoral systems.
💧Water Accessibility
Very high weight. Water scarcity is the most urgent trigger. Water point operational status (from community reports + satellite) is critical.
🌧️Precipitation Trend
High weight. Precipitation deficit drives both vegetation stress and water scarcity — it is the upstream cause of most movement pressure.
⛰️Terrain Feasibility
Moderate weight. Acts as a constraint rather than a driver. Does not create movement pressure but determines which routes are physically viable.
⚠️Risk Zones
Modifier. Active risk zones (conflict, disease, infrastructure failure) reduce classification by one tier if the route passes through them.
Honest Uncertainty Display

The system explicitly shows when it has low confidence. These are not failure states — they are honest outputs. A "Low" with full explanation is more valuable than a confident-looking number with no reasoning.

⚠️ Missing Data Handling
▶ expand
When satellite data is unavailable (cloud cover, sensor delay), the system does not substitute estimated values. Instead, it marks the signal as "Unavailable" and adjusts the classification to reflect increased uncertainty. A corridor cannot be classified as High if a primary signal is missing.
🔄 Conflicting Signal Resolution
▶ expand
When signals conflict (e.g., NDVI indicates pressure to move, but community report says no movement observed), the system does not average them into a misleading middle value. It surfaces the conflict explicitly: "Satellite signals suggest movement pressure, but community reports have not confirmed movement initiation." The user sees the disagreement, not a false consensus.
📅 Temporal Validity
▶ expand
Every classification displays the timestamp of its most recent signal input. A corridor classified as High based on two-week-old data is clearly labeled as stale. Classifications degrade automatically as their input data ages, preventing outdated forecasts from appearing current.
Future ML Integration Plan

The current system uses rule-based environmental scoring. This is intentional — it is transparent, auditable, and does not require historical training data that may not yet exist for this region.

Future Machine Learning Expansion

Once sufficient historical movement data is accumulated, a machine learning model (e.g., gradient-boosted trees or LSTM for temporal patterns) can be trained to refine the rule-based classifications. The ML model would be used to improve the existing logic, not replace it — the rule engine remains as a fallback and sanity check. All ML classifications would still be required to surface the top contributing features (SHAP values), preserving explainability.

Community Reports

Field reports from pastoralists, monitors, and NGO workers. This is the ground-truth layer that confirms or contradicts satellite signals.

Why Community Reports Are Central to This System

Satellites observe the environment. Humans observe the animals. No satellite can see whether a borehole is broken, whether a tribal conflict is blocking a traditional route, or whether movement has already started. Community reporters fill these critical gaps. Crucially, when a community report confirms what satellites predict, the classification moves up. When it contradicts, we surface the conflict rather than hide it.

Technical Pipeline

The full architecture required to make this system operational — from satellite ingestion to corridor visualization.

CURRENT STATE
This prototype implements the frontend layer only. The pipeline below describes what a fully operational version requires. Each layer is a distinct backend component with defined interfaces.
Layer 1 — Data Ingestion
1A

Satellite Data Ingestion

Scheduled fetching from satellite APIs (USGS Earth Explorer, NASA Earthdata, Copernicus). Raw satellite scenes are ingested, cloud-masked, and converted to analysis-ready data products (NDVI, water body extent, soil moisture, precipitation anomaly). Processing via rasterio / GDAL pipeline.

Python · GDAL · rasterio · xarray
1B

Community Report Ingestion

Field reports submitted via SMS gateway (Africa's Talking / Twilio), mobile app (KoboToolbox / ODK), or web form. Reports are parsed, geolocated, timestamped, and stored in a structured database. A moderation queue handles verification before reports influence classifications.

FastAPI · PostgreSQL · Twilio SMS · KoboToolbox
Layer 2 — Spatial Processing
2A

GIS Processing & Accessibility Modeling

Satellite data layers are reprojected to a common coordinate reference system and spatial resolution. A terrain accessibility surface is computed from DEM data, slope analysis, and flood masks. The result is a movement feasibility raster — indicating which areas and routes are physically traversable.

PostGIS · QGIS Server · Least-cost Path Analysis · DEM
2B

Signal Extraction & Normalization

For each candidate corridor zone, signal values are extracted and normalized to a common scale. NDVI anomaly, precipitation deficit, water body change, and slope gradient are each converted into a dimensionless signal strength value (Strong / Moderate / Weak / Unavailable) using defined thresholds from pastoral ecology literature.

NumPy · SciPy · Calibrated thresholds from literature
Layer 3 — Decision Engine
3

Rule-Based Environmental Scoring

The classification engine applies the signal weighting logic to generate a High / Moderate / Low forecast for each corridor. The engine records which signals were available, their strengths, and the exact decision path — producing a full audit trail for every classification. No black boxes: every output is traceable to specific inputs.

Python decision trees · Fully auditable · No ML required (v1) Future: XGBoost · LSTM · SHAP explainability
Layer 4 — Forecast Output
4A

Corridor Visualization API

Computed corridor classifications are served via a REST API as GeoJSON. The frontend consumes this API to render corridor overlays on the Leaflet.js map. Each GeoJSON feature includes classification, signal breakdown, and the decision audit trail as metadata.

FastAPI · GeoJSON · Leaflet.js / Mapbox GL
4B

Alert & Notification System

High-classification corridors approaching risk zones trigger alerts to humanitarian coordinators. Alerts are sent via email, SMS, or integration with existing humanitarian coordination platforms (HDX, ReliefWeb API).

Celery + Redis · Email / SMS · HDX integration
Future: Machine Learning Expansion

The current rule-based engine is the correct starting point. Once 12–24 months of validated movement data has been collected against ground truth, a supervised ML model can be trained to improve classification accuracy — particularly in edge cases where signals conflict.

Planned ML Architecture
  • Input features: same signals used by rule engine + temporal lags + historical corridor usage frequency
  • Model type: Gradient-boosted tree (XGBoost) for tabular signals; LSTM for temporal sequence data
  • Explainability: SHAP values required on every prediction — no unexplained outputs
  • Validation: Holdout against confirmed movement reports; human review of disagreements
  • Deployment: ML model runs alongside rule engine; disagreements are surfaced to analysts
